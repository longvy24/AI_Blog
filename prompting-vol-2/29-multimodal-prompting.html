Why You're Still Typing Essays

Your Screen Has More Signal

Open your recent ChatGPT history and scroll. How many of those prompts start with "on the dashboard, the headline says..." or "in the email it reads..." or "the button is currently labeled..."? That's you, by hand, transcribing pixels into prose. You're describing a thing the model could just look at in one shot.

Until about 2024, that was the only option. Models were text-in, text-out. So we all learned to describe screens like radio narrators describing a movie. Then GPT-4V dropped. Then Claude got vision. Then Gemini. Now every major chat model can take an image as natively as a sentence, and most people are still typing paragraphs because the habit didn't update with the tool.

The cost of typing-instead-of-pasting isn't just time — it's information loss. Your screenshot contains color, spacing, hierarchy, mood, the actual font weight of that headline, the relationship between elements. Your prose contains maybe 10% of that and you have to rebuild it word by word every time you ask.

The Anatomy

Text + Image = Rich Answer

Multimodal prompting is one prompt with two parts. A text instruction and an attached image. That's it. The model reads the text the way it always has, and looks at the image the way you look at a photo — picking up content, layout, tone, color, and relationships in one parallel pass. Then it answers from both.

The trick isn't to write less. It's to describe less and instruct more. You stop saying "the headline says X, the button reads Y, the colors are Z" — because the model can already see all of that. You spend your prompt budget on the part the model can't see: what you want it to do. Rewrite. Critique. Convert. Match the tone. Translate. Build the component.

★ 01 · INSTRUCTION — WHAT TO DO
One sentence. A verb and an object.
Carries the intent. Don't describe what's in the image. Tell the model what to do with it. Examples: rewrite this UI copy · build this as a React component · summarize this whiteboard · convert this sketch into a wireframe.
★ 02 · ATTACHMENT — WHAT TO LOOK AT
A screenshot, a sketch, a photo. The thing.
Carries the context. Drag, drop, paste, or upload. Any modern chat surface accepts PNG, JPG, and most accept PDF. Examples: UI screenshot · napkin wireframe · whiteboard photo · error log image · pricing page screenshot.
// The Universal Multimodal Template
[Verb] this [output type]. [Constraint or tone note]. attach image

Two slots. Verb + output type. Add a constraint when the default isn't what you want (length, tone, format, technology). Then drop the image. That's the whole shape of a multimodal prompt. The image carries all the context that used to take five paragraphs to set up.

Three Use Cases To Steal

Built, Tested, Try Tonight.

Three multimodal prompts I use most weeks. Each one is built around the same shape — short verb, what-to-produce, drop the image. Use them as-is or remix the verb. They cover most of what knowledge workers actually need.

★ 01 · DESIGNER · SCREENSHOT → COPY
The Screenshot Rewrite
When: a UI feels off and you can't articulate why — let the model see it and match the surrounding tone.
"Rewrite the copy on this screen to feel more conversational. Match the warmth of the illustration. Keep the button under 6 words. dashboard-v3.png"
"Headline: 'Good to see you again, Alex.' Button: 'Let's pick up where you left off →' (Matched the sage-green of the illustration. Reduced cold blue accents.)"
★ 02 · BUILDER · SKETCH → CODE
The Sketch-to-Component
When: you've got a napkin wireframe, a whiteboard sketch, or a Figma frame and you want runnable code in one shot — skip the spec doc, drop the picture.
"Build this as a working React component. Tailwind, dark mode, props for the title and metric. sketch.jpg"
"<Card variant='dark'> ... 90 lines of Tailwind React, props typed, ready to drop in. Includes a Storybook example."
★ 03 · EVERYDAY · PHOTO → ANSWER
The Photo-of-Anything
When: a whiteboard you can't transcribe, a textbook page, a fridge full of ingredients, a parking sign in a language you don't speak — point your phone, ask one question.
"3 dinner ideas I can make from what's in here. Rank by protein. fridge.jpg"
"1. Eggs + spinach scramble (32g protein) 2. Chicken thigh + rice 3. Tuna pasta. Top pick: #1 — fastest and the spinach is closest to expiring."
Why It Works

Pixels Carry More Than Tokens

A 1440×900 screenshot is roughly 1.3 million pixels. Even after the vision encoder compresses it down, the model receives a structured representation of layout, color, hierarchy, text, and relationships that would take you somewhere between 200 and 500 written words to approximate. And you'd miss things. You'd describe the headline and forget the button radius. You'd mention the colors and skip the spacing. The image is more reliable than the description because the description is your interpretation of the image, lossy by definition.

The other reason it works: the model is doing less guessing. When you describe a screen in prose, the model has to reconstruct the screen in its head before it can act on your request. That reconstruction is probabilistic and often wrong. When you paste the image, the reconstruction step disappears. The model goes straight to the answer.

Build the Habit

Multimodal Checklist

  • Audit your last 20 prompts — count how many start with "the X says..." or "in this Y...". Every one is a candidate for replacement with a screenshot
  • Make screenshot the default — bind Cmd+Shift+4 (Mac) or Win+Shift+S (Windows) to muscle memory. Most prompts about screens should start with that keystroke, not typing
  • Cut the description, keep the instruction — once the image is attached, delete every sentence that describes what's in it. Keep only what you want the model to do
  • Try one new use case tonight — screenshot a UI, sketch a feature on paper, or photograph something you'd normally Google. Drop it in chat with one line of instruction
  • Use sketches for code — wireframes don't have to be Figma. A photo of a napkin sketch is enough for a working React component. Stop building specs, start building shapes
Common Mistake

Don't Describe and Attach

The most common failure mode is doing both. People paste the image and then write the same five-paragraph essay they would have written without it. The whole point is that the image replaces the essay. If you're describing what's in the picture, you're paying tokens twice for the same context — and worse, you're giving the model two slightly different versions to reconcile. Trust the image. Cut the prose.

The other failure mode is forgetting the instruction. A bare image with no text is "what is this?" which is not usually what you want. The image is the noun. You still have to provide the verb. One sentence. What do you want done.

// The One Thing To Remember

Stop describing your screen. Show it. Two-part prompt — verb plus image. The pixels carry the context that used to take five paragraphs to set up, and the model goes straight to the answer with no reconstruction step in between. Try one tonight: screenshot a UI, drop it in chat, ask for the rewrite. Fifteen seconds beats eight minutes every single time.
