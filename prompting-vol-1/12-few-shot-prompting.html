The Frustration

ChatGPT Won't Stay In Format

You ask for a summary in three bullet points. You get five. You ask for an email in your tone. You get something that sounds like a generic newsletter. You ask for a JSON object with three fields. You get something kind of like JSON with extra commentary at the end. The model heard you. It just didn't commit to the format.

The fix isn't telling it harder. Adding "really, only three bullets" or "JSON only please" works sometimes, fails other times, and feels like bargaining. The reliable fix is showing instead of telling — paste two or three concrete examples of input and the exact output you want, and the model will mirror the structure of your last example almost exactly.

The Fix

Show Two Or Three Examples

The template is one of the cleanest in prompt engineering — it's just a list of input-output pairs followed by your actual task:

// The Few-Shot Prompting Template
Here are examples of the format I want: Input: [example input 1] Output: [example output 1] Input: [example input 2] Output: [example output 2] Input: [example input 3] Output: [example output 3] Now do the same for: Input: [your actual input] Output:

Three things make this template work where "be consistent" prompts fail:

  • The examples define the format more precisely than words can. "Three bullets, each starting with a verb, no longer than ten words" is a sentence. Three actual bullets that do those three things is an unambiguous structural signal the model can mirror exactly. Wispr Flow's prompting guide makes the same point bluntly: "A vague instruction like 'professional but approachable' can mean different things to different people. A concrete example or description provides an actual target."
  • Two examples is the floor, three is the ceiling. One example feels like a coincidence — the model might or might not infer the pattern. Two examples lock in the pattern. Three examples remove all remaining ambiguity. The range is supported by published guidance — DataCamp's tutorial puts the practical sweet spot at "just 2-5 examples", and IBM characterizes few-shot as using "multiple examples to improve accuracy and adaptability" versus the brittleness of one-shot's single example. Beyond three, you're spending tokens without buying much accuracy.
  • The Output: token at the end is doing real work. It tells the model "your turn — produce the same kind of thing." Without that closing token, the model sometimes adds preamble or asks a clarifying question. With it, the next thing it generates is the output.

"One example, good. Three examples, nearly perfect."

Why It Works

In-Context Pattern Matching

Under the hood, few-shot prompting is exploiting a property called in-context learning. DataCamp describes the mechanism as a four-step cascade — "pattern recognition, task inference, generalization, and application" — all happening at inference time without any parameter updates. The transformer reads your example pairs and updates its attention patterns to favor the structural regularities it sees: the line lengths, the punctuation, the leading verbs, the field order. It's not learning in the gradient-descent sense — no weights are changing. It's conditioning on the demonstrations in the same way it conditions on a role prompt, but with a much stronger structural signal because the demonstrations are concrete instead of abstract.

IBM frames the same idea in production terms: few-shot prompting "leverages the pre-trained knowledge of large language models to perform specific tasks efficiently, even with limited data" — the model already knows how to write JSON, how to write professional emails, how to summarize prose. Your examples don't teach those skills; they signal which of those skills to apply and in what specific format.

The reason three examples beats one isn't that the model is bad at generalizing from a single example. It's that one example is ambiguous about what's essential versus accidental. Was the comma after "Output" significant? Was the all-caps "Input" load-bearing? With three examples, the structural features that repeat are clearly the pattern; the features that vary are clearly the content. The model can tell the two apart, and it mirrors the pattern.

The Catch

Bad Examples Become The Format

Few-shot prompting has one failure mode that's worse than the failure mode it fixes: if your examples are inconsistent, the model copies the inconsistency. If example 1 has a period at the end of every bullet and example 2 doesn't, the model will sometimes punctuate and sometimes not — that variance is now part of the pattern, not noise the model filters out. Examples need to be parallel down to the punctuation.

Second catch: examples eat tokens fast. Three full input-output pairs can easily run 200-500 tokens before you've even gotten to your real task. On free-tier models or long conversations, you can blow past the context window with the examples alone. IBM lists this directly among the technique's limitations — "dependence on prompt quality, computational complexity, [and] challenges in generalization" are the trade-offs you accept in exchange for format reliability. The fix is to keep examples short — one-line input, one-line output — unless the format genuinely requires more.

Third catch: the model overfits to surface features. If all three of your examples happen to be about coffee, the model will sometimes assume you're asking about coffee. DataCamp gives the same warning in different words — "use a variety of examples that cover different scenarios and edge cases" — because the alternative is the model treating an accidental topical pattern as part of the requested format. Vary the subject matter of the examples to signal "the pattern is the structure, not the topic."

Real Use Cases

Five Places Few-Shot Earns Its Tokens

Both the creator-facing applications below and the heavier industry use cases (sentiment analysis, named-entity recognition, code-test generation, grounded dialog) IBM documents are the same technique with different example sets. The first five are what V12 is selling; the bracketed enterprise applications are what production teams reach for when the format reliability matters more than the convenience.

  • Matching writing style — paste three sentences in your voice, ask for a paragraph in the same voice. Works on emails, blog intros, captions, anywhere tone matters more than structure. Wispr Flow's guide gives the canonical version of this: "I'm attaching an email I liked the tone of. Write the new one in a similar voice."
  • Structured data extraction — paste three examples of unstructured input (a paragraph) and the JSON object you want extracted, then paste the new paragraph. Beats every "give me JSON" prompt for reliability. Closely related to IBM's documented Named Entity Recognition use case where examples teach the model your taxonomy of entity types.
  • Email templates — three examples of "received email → my preferred reply" turns ChatGPT into a personalized reply engine. The model picks up your length, your tone, your standard closing.
  • Categorization / labeling — three example items with their categories, then a list of new items. The model applies your taxonomy consistently across hundreds of items in one prompt. This is IBM's Sentiment Analysis case in disguise — same technique, different label set.
  • Format transfer — three "before / after" pairs showing the format conversion (markdown → HTML, prose → bullet list, technical → plain language). The model performs the same transform on new input. DataCamp's tutorial walks through this exact pattern for code documentation generation.

Common Mistake

Don't Show Inconsistent Examples

The most common failure mode isn't using too few examples — it's using examples that disagree. If example 1 puts the colon after Input and example 2 puts it after a tab, the model treats both as valid. If example 1 capitalizes the first letter of the output and example 2 doesn't, the model picks one at random for the third. Examples must be structurally identical, varying only in content. Read your three examples side by side before you paste — if you can find any structural difference between them, the model will too.

Second mistake: putting Output: at the start of the line vs. at the end of the input pair. The model is sensitive to that boundary. Pick one, use it across all examples, and end your prompt with the same boundary so the model knows it's its turn.

// Takeaway

Two or three structurally-identical examples, followed by a fresh input and an empty Output: field. The model mirrors the structure of your last example almost exactly. It's the fastest way to get consistent format out of any modern LLM — ChatGPT, Claude, Gemini, all of them.

Pair it with role-based prompting (V11) when stakes are high — role for tone, examples for structure.
